## 1. 总体架构

### 1.1 架构分层
采用 **平台无关核心库 + 平台适配层 + UI 层 + 边缘服务层** 的分层架构，核心目标是让"持仓计算 / 估值 / 回测"等业务逻辑与 Web/小程序解耦，为后续小程序复用打基础。

### 1.2 技术选型
| 层 | 选型 | 理由 |
|----|------|------|
| 前端框架 | React 18 + Vite + TypeScript | 用户指定；生态成熟、组件化、回测交互复杂度适配好 |

| 边缘服务 | Cloudflare Workers + itty-router/Hono | 轻量路由、边缘部署 |

| 构建/部署 | Vite build → Pages；wrangler → Workers | 官方工具链 |
| 测试 | Vitest（核心库 + 单测） | 与 Vite 一体、快 |
| 包管理/Monorepo | pnpm workspace | 管理 core / web / workers 多包 |

## 2. 核心数据模型（@fund/core/domain）

> 所有结构带 `schemaVersion`；金额用「分」或高精度 number（设计采用 number + 统一四舍五入工具，份额保留 4 位，金额保留 2 位）。

## 3. 交易规则引擎（@fund/core/trading）— 决策1

### 3.1 交易日历
- `TradingCalendar`：判定某日期是否交易日、求下一个交易日、计算 T+N。
- A 股交易日历由 Workers `/api/calendar` 提供（含节假日），核心库缓存。
- 离线/无网时退化为"周末非交易日"的近似规则（标注不精确）。

### 3.2 确认日计算
```
计算确认日(submitAt):
  base = submitAt 的日期
  if submitAt.time < 15:00 且 base 是交易日 → T = base
  else → T = base 之后的第一个交易日
  返回 T (作为成交净值取值日 / 确认日)
```

### 3.3 订单状态机
```
下单 → PENDING(等待确认日净值)
   确认日净值可得 → 结算(算份额/金额/费用) → CONFIRMED → 写入 transactions, 更新 positions/cash
   用户撤单(确认前) → CANCELLED
```
- 实时模拟中，若确认日收盘净值尚未公布，订单停留在 PENDING，由"结算服务"在净值可得后回填（前端进入页面/刷新时触发结算检查）。

### 3.4 费用计算
- 申购费（默认外扣）：净申购 = 金额 / (1 + 申购费率)，费用 = 金额 − 净申购；份额 = 净申购 / 净值。
- 赎回费：按 `lots` FIFO 计算每批持有天数，匹配 `redeemFeeTiers` 分档求加权赎回费。
- 转换费：源基金赎回费简化 + 目标申购费，或单一转换费率（按配置）。

### 3.5 持仓与收益计算
- 持仓市值 = 份额 × (估值或最新净值)。
- 持仓收益 = 市值 − 成本；总资产 = 现金 + 在途资金 + Σ市值。
- 当日盈亏 = Σ 份额 × (估值 − 上一交易日净值)。
- 总收益 = 总资产 − 初始总投入(initialCash)；收益率 = 总收益 / 初始总投入。
- **缺价回退（ADJ-3）**：某持仓缺少有效价格（估值未加载 / 数据源失败 / nav≤0）时，
  回退使用其平均成本(cost/shares)作为净值，即按"盈亏 0"处理，避免误算为市值 0 导致的
  −100% 假亏损。前端在构造 priceMap 时跳过 nav≤0 的估值，交由核心库统一回退。

## 4. 估值聚合与多数据源（@fund/core/valuation + workers）— R4/R5/R6

### 4.1 Provider 抽象（核心库定义接口，调用走 API 客户端）
```ts
interface ValuationProvider {
  id: ValuationSourceId;
  name: string;
  getValuation(codes: FundCode[]): Promise<Valuation[]>;
}
```
- 核心库定义接口与聚合逻辑（多源并列、对比、选择）。
- 实际数据获取由 Web/小程序注入的 `ApiClient` 实现（调用 Workers），保持核心库无网络依赖。

### 4.2 接口数据源（Workers 代理）
- `/api/valuation?codes=...&source=eastmoney|danjuan`
- Workers 内 `providers/`：分别适配天天基金、蛋卷的公开估值数据，解析为标准 `Valuation` DTO。
- 缓存：估值数据缓存 30~60s（交易时段），非交易时段更长。
- 限流 + 失败降级（一个源失败自动尝试另一个，前端可显式指定）。

### 4.3 自建"公开持仓估算"数据源 — R6（决策5）
计算流程（`/api/self-nav` + `workers/valuation`）：
```
1. 取基金公开持仓: /api/holdings (季报披露的重仓股/全部持仓 + 各自权重)
2. 取持仓个股实时行情: /api/quote (个股实时涨跌幅)
3. 已覆盖部分估算涨幅 = Σ(个股权重 × 个股实时涨跌幅) / Σ覆盖权重
4. 未覆盖部分(1 - Σ覆盖权重): 按宽基指数(默认沪深300)实时涨跌幅补全
5. 基金估算涨跌幅 = 覆盖部分加权 + 未覆盖部分按指数补全(按各自权重)
6. 估算净值 = 上一交易日净值 × (1 + 估算涨跌幅)
7. confidence = Σ覆盖权重 (覆盖率, 提示用户)
```
- 局限：持仓季度披露有滞后，债券/未披露仓位误差大；输出 `confidence` 标注。
- 缓存：持仓数据缓存较久（天级），个股行情缓存短（秒级）。

### 4.4 估值刷新策略（决策8）
- 默认手动刷新；交易时段可开启自动轮询（默认 60s，可配）。
- 前端聚合批量请求（多基金合并一次调用），降低 Workers 调用量。

### 4.5 估值 vs 实际净值的展示规则（ADJ-4）
持仓展示的"净值/涨跌/当日盈亏"按是否处于"估值变动时段"区分，**对所有数据源统一生效**：
- **判定 `isEstimating()`**：当前为交易日（经交易日历，排除法定节假日）**且**已过 9:15（集合竞价开始）。该窗口覆盖：集合竞价(9:15-9:30)、盘中(含午休)、收盘后——**直至当日净值公布**。
- **盘中（9:15-15:00，含午休）**：今日净值不可能已公布，直接用所选数据源的盘中估值。
- **收盘后（>15:00 至当日净值公布前）**：继续展示估值；逐基金判断——经 `/api/history` 取最新净值点，若其日期 === 今日则视为"已公布"，切换为实际净值，否则继续用估值。
- **估值时段**：列标题"估值 / 估算涨跌"，状态标签"盘中估值"；自建源附覆盖率。
- **非估值时段**（非交易日 / 节假日 / 9:15 前 / 收盘后已公布净值）：经 `/api/history` 取已公布净值与实际当日涨跌，列标题"净值 / 当日涨跌"，状态标签"已公布净值"。（自建源无历史接口，历史回退 eastmoney；选蛋卷时用蛋卷历史。）
- 前端 store 暴露统一的 `DisplayQuote { nav, growthPct, prevNav, isEstimate, source, confidence }` 与 `estimating` 标志（= 当前展示项中是否仍有估值），UI 与盈亏计算据此统一取值。

## 5. 策略与回测引擎（@fund/core/strategy）— R7/R8/R9

### 5.1 策略求值
- 每个策略模板实现 `evaluate(ctx: BacktestDayContext): StrategyAction[]`。
- `ctx` 提供：当前日期、当日及历史净值、当前持仓、可用现金、策略内部状态（如网格层、上次定投日）。

### 5.2 回测引擎
```
输入: StrategySet, 标的(可多只), 区间[start,end], 初始资金, 基准
数据: /api/history 拉取区间历史净值(经 Workers 缓存)
流程(按交易日循环):
  for each 交易日 d in [start,end]:
    1. 结算前一日 PENDING 订单(按场外规则: 上一交易日申报→今日净值确认)
    2. 收集所有策略 evaluate(d) 产生的 actions
    3. 按 conflictPolicy 归并(先卖后买/同向去重)
    4. 生成订单(遵循交易规则引擎: 确认日/费率)
    5. 更新组合状态、记录每日净值/资产快照
输出:
  - 资产净值曲线、收益率、年化、最大回撤、夏普(可选)
  - 操作流水
  - 基准对比(标的买入持有 / 沪深300)
```
- 回测复用 `trading` 引擎的费率/确认逻辑，保证与模拟交易口径一致。
- 性能：大区间在前端 Web Worker 中运行（避免阻塞 UI）；数据批量预取并缓存。

#### 5.2.1 指标与曲线（ADJ-5 / ADJ-6 增强）
- **详尽指标** `BacktestMetrics`：
  - 资金：期初资金、累计买入/卖出回收/费用、累计净投入。
  - 期末状态：期末总资产/现金/持有资产(仅基金市值)/持仓成本。
  - 收益：期末总收益/总收益率/年化/持仓浮盈/持有收益率/持有年化收益率。
  - 风险：总资产最大回撤、持有最大回撤(含峰值/谷底日期)、年化波动率、盈利日占比。
  - 风险调整：夏普比率、索提诺比率、卡玛比率（均基于时间加权持有口径，可配 `riskFreeRate`）。
  - 交易：交易次数(买/卖)、累计费用、交易日数。
  - 基准：买入持有的总收益、年化、最大回撤（与策略对比超额）。
- **每日快照** `DailySnapshot` 增加 `cost`(持仓成本)、`investedCapital`(累计净投入)、`holdingIndex`(时间加权持有指数)。
- **最大回撤口径修复**：原仅有"总资产最大回撤"，含大量闲置现金（如定投场景），会把基金的真实回撤稀释成很小的正数（用户反馈 161725 算出的回撤明显偏小、不正确）。新增**持有最大回撤**：基于**时间加权持有指数**（日因子 = 当日市值 / (上一日市值 + 当日净流入)），剔除现金稀释与定投资金流入，反映持仓本身的真实回撤。实测 161725 在 2021 区间：总资产回撤 4.82%（被现金稀释），持有回撤 31.37%（峰值 2021-02-10→谷底 2021-08-20，与净值 1.6466→1.1255 的真实跌幅吻合）。
- **风险调整比率**：夏普/索提诺/波动率基于持有指数日收益，按 252 交易日年化；卡玛 = 持有年化 / 持有最大回撤。
- **曲线**：前端展示总资产、持仓市值、累计净投入、持仓成本、基准（买入持有）多条曲线。

#### 5.2.2 多策略集横向对比（ADJ-7）
- 回测页采用 Tabs：「单策略集回测」与「多策略集对比」。
- 对比 Tab：多选 2+ 策略集，使用**相同区间/初始资金/费率/基准**各自独立回测（共享一次历史净值拉取，避免重复请求），输出：
  - **指标对比表**：行=指标（期末总资产/总收益率/年化/持有收益率/持有最大回撤/年化波动率/夏普/索提诺/卡玛/盈利日占比/净投入/费用/交易次数），列=各策略集；按"越大/越小越优"用 ★ 高亮每行最优者。
  - **总资产曲线对比图**：每个策略集一条曲线 + 基准曲线（取第一个含基准者），日期轴取最长序列、缺失日按上一有效值延展。

### 5.3 冲突归并（决策4）
```
同一交易日 actions:
  1. 同基金同方向多笔 → 合并(金额/份额相加, 比例取上限)
  2. 同基金既买又卖 → 先执行卖, 再执行买(用卖出回笼现金)
  3. 现金不足时按策略优先级(策略在集合中的顺序)依次满足
```

## 6. 序列化与导入导出（@fund/core/serialization）— R3/R10（决策7）

### 6.1 编码格式
```
导出: object → JSON.stringify → (可选 gzip/pako) → UTF-8 bytes → Base64
导入: Base64 → bytes → (gunzip) → JSON.parse → schema 校验 → 迁移 → 实体
```
- 封装 `encodePortfolio/decodePortfolio`、`encodeStrategySet/decodeStrategySet`。
- **带 magic header + schemaVersion**：如 `FUNDPF1:<base64>` / `FUNDSS1:<base64>`，便于识别类型与版本。
- 解码做严格校验（字段、类型、范围），非法数据抛明确错误，不污染存储。
- 重名冲突（决策7）：导入时若 name 已存在 → 自动追加"(副本)"或序号，生成新 id。

### 6.2 版本迁移
- `migrations[]`：按 schemaVersion 升级旧数据，保证向后兼容。

## 7. 存储抽象（@fund/core/storage）— R11/R12

### 7.1 接口
```ts
interface StorageAdapter {
  get<T>(key: string): T | null;
  set<T>(key: string, value: T): void;
  remove(key: string): void;
  keys(): string[];
}
```
- Web 实现：`LocalStorageAdapter`（packages/web/adapters）。
- 小程序实现（演进期）：`WxStorageAdapter`（`wx.getStorageSync` 等）。
- 核心库只依赖接口，不直接触碰 `localStorage`。

### 7.2 存储键与容量（决策：localStorage ~5MB）
```
fund.portfolios.index         // 持仓集合 id 列表
fund.portfolio.<id>           // 单个持仓集合
fund.strategySets.index
fund.strategySet.<id>
fund.settings                 // 数据源偏好/刷新设置
```
- 回测结果、历史净值等大对象**不长期持久化**（仅运行时/会话内缓存），避免超限。
- 写入前做大小预估，接近上限时提示用户清理/导出。

## 8. Cloudflare Workers API 设计（packages/workers）— R13

### 8.1 路由（Hono / itty-router）
| 方法 | 路径 | 说明 | 缓存 |
|------|------|------|------|
| GET | /api/valuation | 第三方估值(批量, ?codes=&source=) | 30-60s |
| GET | /api/history | 历史净值(?code=&start=&end=) | 数小时-天 |
| GET | /api/holdings | 基金公开持仓(?code=) | 天级 |
| GET | /api/quote | 个股实时行情(?symbols=) | 秒级 |
| GET | /api/self-nav | 自建估值(?codes=) 内部聚合 holdings+quote | 30-60s |
| GET | /api/calendar | 交易日历(?year=) | 长期 |
| GET | /api/fund-info | 基金基础信息/费率(?code=) | 天级 |

### 8.2 设计要点
- **标准化 DTO**：各源差异在 Workers 内消化，前端只见统一结构。
- **缓存**：Cache API（边缘）为主，热点/跨区可选 KV。
- **CORS**：仅允许 Pages 域名 + 开发域名。
- **限流/防滥用**：按 IP/路径简单限流，保护第三方源与自身额度。
- **降级**：源失败返回结构化错误码，前端切换源或提示。
- **合规**：响应头/前端展示"数据来自公开网络、非官方、仅供参考"。
- **Workers 限制对策**：单请求子请求数受限 → 自建估值的个股行情批量合并；大区间历史数据分页/分段缓存。

## 9. 前端设计（packages/web）

### 9.1 页面结构
- **持仓页**：集合切换、总览(总资产/收益)、持仓列表(估值/收益)、操作入口(买入/卖出/转换)、流水、待确认订单。
- **集合管理页**：增删改、导入/导出(Base64 文本框 + 复制/粘贴)。
- **估值页/面板**：多数据源对比、刷新控制、覆盖率(自建源)。
- **策略页**：策略模板配置、策略集编辑、导入导出。
- **回测页**：选策略集/标的/区间/初始资金 → 运行 → 收益曲线/指标/流水/基准对比(ECharts)。

### 9.2 状态管理（Zustand）
- `portfolioStore`、`strategyStore`、`valuationStore`、`settingsStore`。
- store 调用 `@fund/core` 计算，调用 `api` 取数，调用 `adapter` 持久化。

### 9.3 回测放入 Web Worker
- 大区间逐日模拟在独立 Web Worker 执行，避免阻塞 UI；核心库为纯 TS，可直接在 Worker 中运行。

## 12. 关键风险与对策（承接需求 5.3）
| 风险 | 对策 |
|------|------|
| 第三方源变动/受限 | 多源适配 + 自动降级；Provider 解析隔离，便于快速修复 |
| 公开持仓滞后/误差 | 输出 confidence；文案提示；指数补全未覆盖仓位 |
| Workers 额度/限制 | 边缘缓存、批量合并请求、历史数据分段缓存 |
| localStorage 超限 | 大对象不落地、容量预警、导出备份 |
| 计算口径不一致 | 模拟与回测共用 trading 引擎，统一费率/确认逻辑 |
| 小程序演进成本 | 核心库零平台依赖、存储/网络走适配层，UI 可单独实现 |

## 13. 数据流总览（端到端）
```
[用户操作买入] → web store → core.trading 生成 Order(PENDING)
   → adapter 持久化 → (确认日净值可得) core 结算 → 更新 Portfolio → 持久化

[查看估值] → web valuationStore → api → Workers /api/valuation|self-nav
   → 标准化 Valuation → core 聚合多源 → UI 展示对比

[回测] → web → core.strategy 回测引擎(Web Worker)
   → 需要历史数据时 → api → Workers /api/history(缓存) → 逐日模拟 → 指标/曲线

[导入导出] → core.serialization (Base64 + schema) ↔ 文本框
```
